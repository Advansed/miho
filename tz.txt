# Техническое Задание
## Веб-система управления фотосессиями и продажей материалов

**Версия:** 1.0  
**Дата создания:** 07.01.2026  
**Статус:** Черновик

---

## 1. Общие сведения о проекте

### 1.1 Цель проекта
Разработать веб-систему для фотостудии, обеспечивающую:
- Регистрацию клиентов по номеру телефона
- Создание и управление заказами на фотосессии
- Организацию работы операторов и фотографов
- Управление фотосессиями (общие и индивидуальные)
- Загрузку и доступ к отснятому материалу после оплаты через эквайринг
- Автоматическое архивирование материала по сроку хранения

### 1.2 Область применения
Веб-приложение для операционной системы фотостудии малого/среднего размера с поддержкой одновременной работы нескольких фотографов и операторов.

### 1.3 Основные участники
- **Пользователь (Клиент)** — создаёт заказ на фотосессию, оплачивает, получает доступ к материалам
- **Оператор** — обрабатывает заказы, создаёт фотосессии, назначает фотографов
- **Фотограф** — работает в день съёмки, создаёт индивидуальные фотосессии, загружает материалы
- **Администратор** — управляет системой, настройками, пользователями

### 2.2 Описание ролей

#### Пользователь (Клиент)
- Регистрация по номеру телефона с подтверждением кодом.
- Создание заказа на фотосессию (выбор типа, даты, локации, комментарии).
- Просмотр собственного списка общих фотосессий и их статусов.
- Просмотр платёжного задания и оплата через эквайринг.
- После оплаты: просмотр превью и скачивание исходных материалов из личного кабинета.
- Доступ к материалам ограничен по сроку хранения (3–12 месяцев).

#### Оператор
- Авторизация по логину/пароля или по номеру телефона.
- Просмотр всех новых, текущих и завершённых заказов от пользователей.
- Связь с пользователем (вне системы: звонок, мессенджер) для уточнений.
- Создание общей фотосессии: установка даты, времени, места, назначение фотографа.
- Редактирование данных фотосессии, изменение статусов.
- Просмотр расписания по фотографам.
- Управление статусами заказов и сессий (новый → запланирована → завершена → архив).
- Доступ к полному списку платежей (статусы, но не реквизиты карт).

#### Фотограф
- Авторизация по логину/пароля или по номеру телефона.
- Просмотр своего расписания: список назначенных общих фотосессий с деталями (дата, место, контакт основного клиента).
- Внутри каждой общей фотосессии: создание индивидуальных фотосессий для отдельных людей/семей.
- Ввод номера телефона для каждой индивидуальной сессии; система автоматически создаёт/находит аккаунт.
- Загрузка медиафайлов (фото, видео) по общей и индивидуальным фотосессиям.
- Просмотр статусов оплат по своим сессиям (только информация о том, оплачено/не оплачено).

#### Администратор
- Полный доступ ко всем функциям системы.
- Управление пользователями: создание, удаление, блокировка, смена ролей.
- Управление справочниками: типы съёмок, локации, фотографы.
- Настройка сроков хранения материалов (по умолчанию 3–12 месяцев).
- Настройка параметров эквайринга (провайдер, комиссия, реквизиты).
- Просмотр логов операций, отчётов по платежам и сессиям.
- Управление архивом: удаление или перемещение старых материалов.

## 3. Функциональные требования

### 3.1 Модуль авторизации и регистрации

#### 3.1.1 Регистрация пользователя по телефону
- Пользователь вводит номер телефона (формат: +7 (XXX) XXX-XX-XX или аналог).
- Система отправляет SMS с одноразовым кодом (6 символов, срок действия 10 минут).
- При первом входе по телефону автоматически создаётся аккаунт с минимальными данными (телефон, role = 'user').
- При повторных входах используется существующий аккаунт.

#### 3.1.2 Авторизация
- Для пользователей: телефон + код из SMS.
- Для операторов и фотографов: email/телефон + пароль (или телефон + SMS-код).
- Система выдаёт JWT-токен со сроком действия 24 часа (или сессионный cookie).

#### 3.1.3 Восстановление доступа
- По номеру телефона: отправка SMS-кода.
- По email (для операторов/фотографов): ссылка на сброс пароля.

### 3.2 Модуль заказов

#### 3.2.1 Создание заказа пользователем
- Форма с полями:
  - Тип съёмки (dropdown: семейная, детская, love-story, портреты, корпоративные и т.д.).
  - Желаемая дата и диапазон дат.
  - Локация (студия, выезд, свободный выбор).
  - Количество участников / примечание.
  - Контактные данные (уже заполнены из профиля).
  - Дополнительные комментарии.
- При отправке создаётся сущность Order со статусом 'новый'.
- Уведомление отправляется оператору (email или внутренняя панель).

#### 3.2.2 Управление заказом оператором
- Оператор видит список всех заказов с фильтрацией по статусам: новый, согласован, запланирован, отменён.
- Оператор может:
  - Просмотреть детали заказа.
  - Добавить комментарий (для внутреннего использования).
  - Изменить статус заказа.
  - Привязать заказ к созданной фотосессии.

### 3.3 Модуль фотосессий (общие и индивидуальные)

#### 3.3.1 Создание общей фотосессии (оператор)
- Форма с полями:
  - Привязка к заказу пользователя (dropdown).
  - Дата и время начала.
  - Дата и время окончания (опционально).
  - Место проведения (зал студии, выезд).
  - Назначенный фотограф (dropdown).
  - Описание / примечания.
  - Статус (запланирована, завершена, отменена).
  - Срок хранения материалов (по умолчанию из общих настроек).
- При создании фотосессии:
  - генерируется уникальный ID.
  - привязывается к пользователю (владельцу заказа).
  - статус устанавливается в 'запланирована'.

#### 3.3.2 Просмотр расписания фотографом
- Личный кабинет фотографа отображает:
  - Список всех назначенных ему общих фотосессий.
  - Дата, время, место, основной клиент (только ФИ/телефон).
  - Статус сессии.
  - Кнопка для перехода к сессии и создания индивидуальных фотосессий.

#### 3.3.3 Создание индивидуальной фотосессии (фотограф)
- Внутри общей фотосессии фотограф может создать одну или несколько индивидуальных фотосессий.
- Форма с полями:
  - Номер телефона клиента (обязательно).
  - Тип пакета / описание съёмки (опционально).
  - Ориентировочная стоимость (опционально).
  - Комментарии.
- При сохранении:
  - Система проверяет, существует ли аккаунт по данному телефону.
  - Если нет → создаёт новый аккаунт (role = 'user') с phone и статусом 'active'.
  - Если да → использует существующий аккаунт.
  - Индивидуальная фотосессия привязывается к этому аккаунту и к общей фотосессии.
  - Система отправляет SMS/мессенджер-уведомление на номер телефона (опционально).

#### 3.3.4 Управление сессией (Пользователь)
- Пользователь видит список своих фотосессий (общих и индивидуальных, на которых он участвует).
- Для каждой сессии отображаются:
  - Дата, место, фотограф.
  - Статус (запланирована, завершена, ожидает оплаты, оплачена, в архиве).
  - Сроки доступа к материалам.

### 3.4 Модуль медиафайлов

#### 3.4.1 Загрузка материалов (фотограф)
- Фотограф загружает файлы (jpeg, png, raw, mp4 и т.д.) по завершении съёмки.
- Интерфейс:
  - Выбор общей или индивидуальной фотосессии.
  - Drag-and-drop или стандартная загрузка файлов.
  - Опциональные теги / описание для каждого файла.
- При загрузке:
  - Система сохраняет оригинальный файл в хранилище.
  - Автоматически создаёт превью (уменьшенное изображение, водяной знак).
  - Логирует дату и время загрузки, размер.

#### 3.4.2 Доступ к материалам (пользователь)
- До оплаты: доступны только превью с водяным знаком.
- После оплаты: доступны исходные файлы в полном разрешении.
- Доступ ограничен по сроку хранения (указан в сессии; по умолчанию 3–12 месяцев от даты завершения).
- Пользователь может скачать материалы целиком (архив .zip) или по отдельности.

#### 3.4.3 Управление файлами (администратор)
- Просмотр всех загруженных файлов, размеров, сроков хранения.
- Удаление файлов при необходимости (например, в случае удаления сессии).

### 3.5 Модуль платежей и эквайринга

#### 3.5.1 Создание платёжного задания
- После завершения общей или индивидуальной фотосессии фотограф (или автоматически система) создаёт платёжное задание.
- Поля платежа:
  - Сумма (в рублях или др. валюта).
  - Валюта.
  - Описание пакета (например, «Фотосессия: семейная, 20 фото»).
  - ID сессии.
  - ID пользователя-плательщика.
  - Статус (ожидает_оплаты, в_обработке, оплачен, отклонен).
  - Дата создания.
  - Дата платежа (заполняется после успешного платежа).
  - Идентификатор платежа в системе эквайринга (payment_provider_id).

#### 3.5.2 Инициирование платежа (пользователь)
- Пользователь в личном кабинете открывает сессию и видит платёжное задание.
- Нажимает кнопку «Оплатить».
- Система перенаправляет на страницу платёжного сервиса (или открывает виджет эквайринга):
  - Stripe, YuKassa (ЮКасса), Сбербанк AcquiringSolution и т.д.
  - URL с параметрами: amount, currency, order_id, description.
  - Return URL для успешного платежа и для отмены.

#### 3.5.3 Обработка результата платежа
- Система настраивается на получение webhook'ов от провайдера эквайринга.
- При успешном платеже (webhook с статусом 'success' или 'paid'):
  - Обновляется статус платежа в БД на 'оплачен'.
  - Обновляется статус сессии на 'доступ_открыт'.
  - Для всех медиафайлов в этой сессии устанавливается флаг is_available_fullres = true.
  - Отправляется уведомление пользователю (email/SMS): «Оплата успешна, материалы готовы к скачиванию».
  - Логируется транзакция: дата, сумма, ID платежа от провайдера.
- При отклонении или ошибке платежа:
  - Статус остаётся 'ожидает_оплаты' или меняется на 'отклонен'.
  - Отправляется уведомление пользователю: «Платёж не прошёл, свяжитесь с оператором».

#### 3.5.4 Защита от дублирования платежей
- Каждый webhook обрабатывается один раз (проверка по payment_provider_id и статусу в БД).
- Повторные вебхуки игнорируются.

### 3.6 Модуль архивирования и удаления

#### 3.6.1 Сроки хранения
- Для каждой фотосессии в БД хранится поле storage_until (дата и время окончания доступа).
- Срок рассчитывается как (дата_завершения_сессии + настраиваемое_количество_дней).
- По умолчанию: 3–12 месяцев, настраивается в админ-панели для всей системы или для конкретной сессии.

#### 3.6.2 Фоновый процесс архивирования
- По крону (например, каждый день в 00:00 UTC):
  - Проверяются все фотосессии, у которых storage_until < текущее время.
  - Для таких сессий:
    - Статус меняется на 'архив'.
    - Медиафайлы перемещаются в архивное хранилище (отдельное или же помечаются флагом is_archived = true).
    - Пользователь теряет доступ к скачиванию (в интерфейсе выводится сообщение «Срок доступа истёк»).
    - Опционально: удаляются физические файлы с основного хранилища.

#### 3.6.3 Интерфейс пользователя
- В личном кабинете фотосессии разделены на:
  - Активные (с доступом к материалам).
  - Архивные (без доступа, но видны в истории).
- Для архивных сессий информация сохраняется в метаданных, но файлы недоступны.

## 4. Нефункциональные требования

### 4.1 Безопасность
- Все передачи данных по HTTPS.
- Авторизация через JWT-токены (с проверкой подписи и срока действия) или сессионные cookies.
- Защита от CSRF: использование CSRF-токенов в формах.
- Валидация и санитизация всех входных данных (защита от SQL-injection, XSS).
- Хеширование паролей (bcrypt или аналог).
- Ограничение доступа к API по IP или роли.
- Логирование всех критических операций (создание/удаление платежа, загрузка файлов).
- Соответствие GDPR/РФ ФЗ-152 по хранению персональных данных.
- PCI DSS: не хранить реквизиты платёжных карт в системе (всё делегировать эквайрингу).

### 4.2 Производительность
- Время загрузки главной страницы: < 2 сек (на скорости 3G).
- Время загрузки личного кабинета: < 1 сек.
- Оптимизация изображений: автоматическое создание множественных размеров (thumbnail 200x200, preview 800x600, fullres).
- Кеширование превью (HTTP cache headers, CDN по возможности).
- БД индексы: на полях phone, created_at, session_id, user_id, status.
- Поддержка одновременной работы 100+ пользователей без деградации.

### 4.3 Надёжность и отказоустойчивость
- Резервное копирование БД ежедневно.
- Резервное копирование медиафайлов ежедневно (на отдельный диск/облако).
- Обработка отказов эквайринга: если webhook не пришёл, возможность ручной обработки в админ-панели.
- Логирование всех ошибок в лог-файл или систему мониторинга (sentry, logstash и т.д.).
- Retry механизм для отправки SMS и email (до 3 попыток с интервалом).

### 4.4 Масштабируемость
- Архитектура: фронтенд (SPA) + backend API (REST или GraphQL).
- Возможность горизонтального масштабирования backend (stateless design).
- Разделение хранилища файлов: основное (для активных сессий) и архивное (для старых).
- Использование queue-системы (например, Celery, RabbitMQ) для фоновых задач (отправка SMS, архивирование).

### 4.5 Совместимость
- Браузеры: Chrome, Firefox, Safari, Edge (последние 2 версии).
- Мобильные: iOS Safari, Android Chrome.
- Адаптивный дизайн (mobile-first подход).

## 5. Модель данных

### 5.1 Таблица User
```
id (UUID)
phone (VARCHAR, UNIQUE)
email (VARCHAR, NULLABLE, UNIQUE)
first_name (VARCHAR, NULLABLE)
last_name (VARCHAR, NULLABLE)
role (ENUM: 'user', 'operator', 'photographer', 'admin')
password_hash (VARCHAR, NULLABLE)
created_at (TIMESTAMP)
updated_at (TIMESTAMP)
deleted_at (TIMESTAMP, SOFT DELETE, NULLABLE)
status (ENUM: 'active', 'blocked', 'pending_confirmation')
```

### 5.2 Таблица Order
```
id (UUID)
user_id (UUID, FK -> User)
order_type (VARCHAR: 'family', 'children', 'love_story', 'corporate', etc.)
preferred_date_from (DATE)
preferred_date_to (DATE)
location_request (VARCHAR, NULLABLE)
participants_count (INT, NULLABLE)
comment (TEXT, NULLABLE)
status (ENUM: 'new', 'agreed', 'scheduled', 'cancelled')
created_at (TIMESTAMP)
updated_at (TIMESTAMP)
```

### 5.3 Таблица Session (фотосессия)
```
id (UUID)
order_id (UUID, FK -> Order, NULLABLE)
user_id (UUID, FK -> User) — владелец заказа
photographer_id (UUID, FK -> User) — фотограф
session_type (VARCHAR: 'general', 'individual')
```

## 6. API Endpoints

### 6.1 Авторизация

#### POST /api/auth/register
Регистрация по телефону.
```
Request:
{
  "phone": "+7 (999) 123-45-67"
}

Response (200):
{
  "status": "success",
  "message": "SMS отправлена на номер +7999..."
}
```

#### POST /api/auth/verify
Подтверждение кода из SMS.
```
Request:
{
  "phone": "+7 (999) 123-45-67",
  "code": "123456"
}

Response (200):
{
  "status": "success",
  "token": "eyJhbGc...",
  "user": {
    "id": "uuid",
    "phone": "+7999...",
    "role": "user"
  }
}
```

#### POST /api/auth/logout
Выход из системы.
```
Response (200):
{
  "status": "success",
  "message": "Вы вышли из системы"
}
```

### 6.2 Заказы

#### POST /api/orders
Создание заказа.
```
Request (требует авторизацию):
{
  "order_type": "family",
  "preferred_date_from": "2026-02-01",
  "preferred_date_to": "2026-02-07",
  "location_request": "студия",
  "participants_count": 4,
  "comment": "Хотим цветной фон"
}

Response (201):
{
  "status": "success",
  "order": {
    "id": "uuid",
    "user_id": "uuid",
    "status": "new",
    "created_at": "2026-01-07T12:00:00Z"
  }
}
```

#### GET /api/orders
Список заказов (для оператора/админа).
```
Response (200):
{
  "status": "success",
  "orders": [...]
}
```

#### GET /api/orders/:id
Детали заказа.

#### PATCH /api/orders/:id
Редактирование заказа.

Статус платежа.

## 7. Интеграции

### 7.1 SMS-сервис
- Выбранный провайдер: (например, Twilio, Exolve, или локальный)
- Использование для: отправка кодов подтверждения, уведомления о статусах.

### 7.2 Эквайринг
- Основной провайдер (выбор): Stripe, YuKassa, Сбербанк Acquiring, VTB и т.д.
- Методы платежа: карты (Visa, MasterCard, Мир), электронные кошельки.
- Интеграция: REST API / виджет на странице.
- Webhook-обработка: получение уведомлений о результатах платежа.

### 7.3 Email-сервис (опционально)
- Отправка уведомлений об оплате, истечении доступа.
- Провайдер: SendGrid, Mailgun, или локальный SMTP.

### 7.4 Хранилище файлов
- Опция 1: локальное файловое хранилище на сервере.
- Опция 2: облачное хранилище (Yandex Cloud, VK Cloud, AWS S3).
- Требование: доступ по HTTPS, возможность создания временных ссылок для скачивания.

## 8. Требования к платформе и инструментам

### 8.1 Backend
- Язык: Node.js (JavaScript/TypeScript) или Python (Django/FastAPI).
- Фреймворк: Express.js / NestJS (Node.js) или Django (Python).
- БД: PostgreSQL или MySQL.
- ORM: Sequelize, TypeORM, SQLAlchemy.
- Кеширование: Redis (опционально).
- Queue: Celery (Python) или BullMQ (Node.js) для фоновых задач.

### 8.2 Frontend
- Фреймворк: React или Vue.js (с TypeScript).
- Управление состоянием: Redux или Vuex.
- HTTP клиент: axios или fetch API.
- UI компоненты: Material-UI, Ant Design, или собственные.
- Адаптивная верстка: CSS Grid, Flexbox, media queries.

### 8.3 Хостинг и DevOps
- Сервер: Linux (Ubuntu, AlmaLinux, AstraLinux).
- Контейнеризация: Docker (опционально).
- CI/CD: GitHub Actions, GitLab CI или Jenkins.
- Мониторинг: Prometheus + Grafana, или Sentry для логирования ошибок.
- SSL: Let's Encrypt или платный сертификат.

## 9. Этапы разработки

### Этап 1: Инфраструктура и базовая авторизация (2–3 недели)
- Настройка серверного окружения, БД, хранилища файлов.
- Реализация регистрации и авторизации по телефону.
- Базовая структура frontend (главная, личный кабинет).

### Этап 2: Модуль заказов и фотосессий (2–3 недели)
- Создание и управление заказами (пользователь).
- Создание общих фотосессий (оператор).
- Интерфейс расписания для фотографа.

### Этап 3: Индивидуальные фотосессии и загрузка контента (2 недели)
- Создание индивидуальных фотосессий (фотограф).
- Загрузка медиафайлов, автоматическое создание превью.
- Интерфейс просмотра материалов (для пользователя).

### Этап 4: Интеграция платежей (2 недели)
- Интеграция с эквайрингом (Stripe/YuKassa/Сбербанк).
- Обработка webhook'ов, изменение статусов.
- Страница оплаты и уведомления.

### Этап 5: Архивирование и администрирование (1–2 недели)
- Фоновый процесс архивирования по срокам.
- Админ-панель для управления системой.
- Отчёты и логирование.

### Этап 6: Тестирование и оптимизация (1–2 недели)
- Unit-тесты, интеграционные тесты.
- Тестирование безопасности (OWASP Top 10).
- Оптимизация производительности.

### Этап 7: Запуск в production (1 неделя)
- Развёртывание, настройка мониторинга.
- Обучение операторов и фотографов.
- Go-live и поддержка.

## 10. Критерии приёмки

- ✓ Все пользователи могут регистрироваться по телефону без ошибок.
- ✓ Заказы корректно создаются, редактируются, привязываются к сессиям.
- ✓ Фотографы видят свой расписание и могут создавать индивидуальные фотосессии.
- ✓ Материалы загружаются, превью доступны до оплаты.
- ✓ Платежи обрабатываются корректно, доступ открывается после оплаты.
- ✓ Сроки хранения работают, архивирование происходит автоматически.
- ✓ Система нагружается 100+ одновременными пользователями без критических ошибок.
- ✓ Все API endpoints протестированы и документированы.
- ✓ Бекап БД и файлов работает без сбоев.
- ✓ Администратор может управлять пользователями и настройками.
